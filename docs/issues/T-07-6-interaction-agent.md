# Description

**Interaction Agent (A2A Orchestrator)**

A dedicated LangGraph agent that owns the user-facing experience: conversation flow, A2UI rendering, and voice orchestration. It communicates with specialist agents (analysis, recommendation, pipeline) via A2A (Agent-to-Agent protocol), making agent-to-agent communication a visible, core architectural element — not just a feature we talk about.

## Role

The Interaction Agent is the **only agent the user talks to**. It:

- Manages conversation state and flow (greeting, clarifying questions, presenting results)
- Sends A2UI messages to control the frontend (logo animation, dashboard embedding, voice controls)
- Delegates domain tasks to specialist agents via A2A
- Translates specialist agent responses into user-friendly A2UI presentations
- Orchestrates the voice pipeline (routes STT output as input, sends responses to TTS)

It does **not** contain domain logic (use-case analysis, method recommendation, dataset search). That stays in the specialist agents from US-03/US-04.

## Architecture

```
                    A2UI (SSE/WS)
                         |
               +--------------------+
               | Interaction Agent  |
               |   (LangGraph)     |
               +----+---------+----+
                    |         |
                   A2A       A2A
                    |         |
         +----------+    +----+----------+
         | Analysis  |    | Recommendation|    ...
         | Agent     |    | Agent         |
         +-----------+    +---------------+
```

## A2A Integration

- Use Google's A2A protocol for inter-agent communication
- The Interaction Agent acts as A2A client, specialist agents as A2A servers
- Each specialist agent exposes an A2A-compatible endpoint (Agent Card with capabilities)
- The Interaction Agent discovers and invokes specialist agents based on conversation context
- A2A task lifecycle: send task → stream updates → receive result

## Relationship to Existing Agents (US-03)

The current monolithic agent in `agent/` handles both conversation and domain logic. This task refactors the architecture:

- **Before**: Single LangGraph agent does everything (conversation + analysis + recommendation)
- **After**: Interaction Agent handles conversation/UI, delegates domain work via A2A

The existing agent logic from US-03 (analysis, recommendation, decision tree) is preserved — it moves into specialist agents that the Interaction Agent calls via A2A. The LangGraph state machine in `agent/graph.py` is split:

- States related to **user interaction** (GREETING, conversation management) → Interaction Agent
- States related to **domain work** (ANALYSIS, RECOMMENDATION, CONFIGURATION, EXECUTION) → specialist agents

## A2UI Message Generation

The Interaction Agent generates A2UI messages to control the frontend:

- **Processing state**: Sends `soofi-logo` component update (idle → processing) when delegating to a specialist agent, back to idle when the response arrives
- **Results presentation**: Translates specialist agent output into A2UI cards, buttons, structured layouts
- **Dashboard triggers**: Sends `dashboard-embed` messages when context calls for it (T-07-5)
- **Voice flow**: Coordinates with STT/TTS services and sends `voice-controls` state updates

## Demo Value

This architecture makes A2A visible and tangible:

- obs-api (T-07-5) can visualize the agent-to-agent communication in real-time
- The logo pulses while specialist agents are working — the user sees the delegation happening
- Langfuse traces show the full chain: user → Interaction Agent → A2A → specialist → response
- On Hannover Messe, we can explain and demonstrate A2A live, not just claim we use it

## Acceptance Criteria

- [ ] A2UI messages are generated by the Interaction Agent (not specialist agents)

- [ ] Domain logic (analysis, recommendation) delegated to specialist agents via A2A
- [ ] Logo animation state reflects A2A task lifecycle (idle → processing → idle)
- [ ] Voice pipeline routed through the Interaction Agent
